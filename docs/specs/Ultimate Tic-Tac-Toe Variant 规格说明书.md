# 终极井字棋变体（Ultimate Tic-Tac-Toe Variant）游戏规格说明书

## 一、项目概述

### 1.1 项目目标

支持双人联机和单机的终极井字棋变体游戏，提供流畅的Web端游戏体验。

### 1.2 项目背景

终极井字棋是一种进阶版的井字棋游戏，通过9x9的大棋盘和9个3x3的小棋盘的嵌套结构，极大地增加了游戏的策略深度和复杂度。

## 二、游戏规则（详细版）

### 2.1 棋盘结构

- **大棋盘**：9×9的网格区域
- **小棋盘**：大棋盘内部平均分为9个3×3的小棋盘
- **小棋盘编号**：从左上至右下依次编号为0-8

```
大棋盘结构：
+---------+---------+---------+
| 小棋盘0 | 小棋盘1 | 小棋盘2 |
|  (0-8)  |  (0-8)  |  (0-8)  |
+---------+---------+---------+
| 小棋盘3 | 小棋盘4 | 小棋盘5 |
|  (0-8)  |  (0-8)  |  (0-8)  |
+---------+---------+---------+
| 小棋盘6 | 小棋盘7 | 小棋盘8 |
|  (0-8)  |  (0-8)  |  (0-8)  |
+---------+---------+---------+
```

### 2.2 玩家与棋子

- **玩家数量**：2名玩家
- **先手**：执黑棋
- **后手**：执白棋
- **棋子样式**：参考围棋云子（黑白两色）

### 2.3 行棋规则

#### 2.3.1 第一手棋
- **无限制**：玩家可以在任意小棋盘的任意位置落子

#### 2.3.2 第二手棋起
- **限制规则**：每一手棋必须下在上一手棋在其小棋盘所在的相对位置在大棋盘上的投影小棋盘里
- **通俗解释**：上一手落在小棋盘X的位置Y，下一手必须落在小棋盘Y内
- **示例**：
  - 上一手：落在小棋盘3的位置7 → 下一手必须在小棋盘7内落子
  - 上一手：落在小棋盘4的位置2 → 下一手必须在小棋盘2内落子

#### 2.3.3 自由落子规则
- **触发条件**：当上一手棋所在的小棋盘编号在大棋盘上的投影小棋盘已被占领时
- **自由范围**：可在所有未被占领的小棋盘内任意位置落子

#### 2.3.4 小棋盘被占领规则
- **占领条件**：当一个小棋盘里某一方完成三连珠（横、竖、斜任意方向）后
- **占领效果**：
  - 该小棋盘由获胜方占领
  - 占领后该小棋盘禁止任何行棋
  - 结算时算作占领方1个子
- **禁止行棋**：被占领的小棋盘在任何情况下都不允许落子

### 2.4 结算条件

#### 2.4.1 常规结算
- **条件a**：当所有未被占领的小棋盘被下满（无空位）时，游戏自动进行结算

#### 2.4.2 主动结算
- **条件b**：当所有未被占领的小棋盘无法决出胜方时（即每个未占领的小棋盘都是平局状态），游戏的任何一方可在其行棋时选择进行结算
- **触发方式**：点击结算按钮

### 2.5 胜负判定

- **胜利条件**：占领小棋盘数量更多的一方获胜
- **失败条件**：占领小棋盘数量较少的一方失败
- **平局条件**：双方占领小棋盘数量相等

### 2.6 附加规则

#### 2.6.1 认输
- **时机**：游戏过程中可随时认输
- **效果**：立即判定当前对手获胜

#### 2.6.2 求和
- **限制**：一局游戏内每方最多可发起3次和棋请求
- **流程**：
  - 玩家A发起求和 → 玩家B响应（接受/拒绝）
  - 若玩家B超时未响应，可再次发起
- **拒绝处理**：对方拒绝后可继续发起，最多3次

#### 2.6.3 超时设置（建议）
- **行棋超时**：单个回合最长思考时间（如60秒）
- **响应超时**：和棋请求等待时间（如30秒）

## 五、前端设计要求

### 5.1 视觉风格

- **参考对象**：macOS自带国际象棋游戏
- **游戏视角**：俯视视角
- **整体风格**：偏3D游戏风格，真实感强

- **棋盘材质**：真实木质或石质纹理

- **网格线条**：清晰但不过于突兀

- **样式**：围棋云子（扁圆形）
- **材质**：半透明玉质感
- **光影**：有光泽和阴影效果

### 5.4 特效要求

- **光追效果**：环境光反射
- **阴影**：棋子投射阴影
- **高光**：棋子表面高光

### 5.5 动画与音效

- **落子动画**：平滑过渡，带轻微反弹
- **落子音效**：清脆的棋子声
- **胜利音效**：胜利时播放庆祝音效

## 十一、规则判定补充说明（新增）

### 11.1 小棋盘状态定义

每个小棋盘在任一时刻应处于以下三种状态之一：

| 状态 | 含义 | 是否允许继续落子 |
|------|------|------------------|
| 未占领且可继续对局 | 尚未分出胜负，且仍存在形成三连珠的可能 | 是 |
| 未占领但已成死局 | 尚未分出胜负，但无论如何都不可能再形成三连珠 | 否（若规则层已判定为不可再落则不允许；若UI仍显示空位，也不得视为可继续争胜） |
| 已被占领 | 某一方已完成三连珠并占领该小棋盘 | 否 |

> 注：当前实现中，死局小棋盘的核心意义是“可用于主动结算判定”。如果未来产品决定允许玩家继续在死局小棋盘内补满落子，则需要明确这不会改变“该盘已无胜方可能”的结论。

### 11.2 合法落子区域判定顺序

每回合开始时，系统应按以下顺序确定合法落子区域：

1. 若当前为首手，则所有未被占领且仍可落子的位置均合法。
2. 否则，读取上一手落子在小棋盘内的位置编号 `cellIndex`。
3. 令目标小棋盘编号 = `cellIndex`。
4. 若目标小棋盘未被占领，且仍存在允许落子的格位，则本回合只能在该小棋盘内落子。
5. 若目标小棋盘已被占领，或虽未被占领但已不可继续有效落子，则进入自由落子。
6. 自由落子时，只能在所有满足“未被占领且允许落子”的小棋盘内选择位置。

### 11.3 小棋盘胜负与计分关系

- 小棋盘一旦被某方占领，计入该方 **1 个占领子**。
- 单个小棋盘不会重复计分。
- 大棋盘层面**不再额外判断三连珠整盘胜利**，最终结果仅依据“占领小棋盘数量”比较。

### 11.4 认输、求和、结算的优先级建议

为避免联机模式下的竞态条件，建议优先级如下：

1. **认输**：最高优先级，触发后立即结束对局。
2. **接受求和**：一旦接受，立即结束对局并记为平局。
3. **主动结算**：仅在满足主动结算条件时允许触发。
4. **普通落子**：仅在对局未结束且未被上述事件打断时允许执行。

## 十二、状态机与交互约束（新增）

### 12.1 对局状态建议

| 状态 | 说明 |
|------|------|
| waiting_start | 对局尚未开始 |
| playing | 正常行棋中 |
| draw_offer_pending | 存在待响应的求和请求 |
| settled | 已进入结算并显示结果 |
| finished | 对局完全结束，不可再操作 |

### 12.2 前端必须明确呈现的信息

每个时刻前端至少应能明确展示以下信息：

- 当前轮到哪一方行棋。
- 上一手落子所在的小棋盘编号和位置编号。
- 当前目标小棋盘编号，或“当前为自由落子”。
- 当前双方已占领的小棋盘数量。
- 当前是否存在待响应的求和请求。
- 当前是否满足主动结算条件。

### 12.3 禁止的前端交互行为

- 不允许点击已被占领的小棋盘中的任意格位。
- 不允许点击非当前合法区域中的格位。
- 不允许在对局结束后继续落子。
- 不允许在单方 3 次求和次数耗尽后继续发起求和。
- 不允许在已有待处理求和请求时重复发起新的求和请求。

## 十三、联机模式服务端约束（新增）

> 本章节为后续联机实现的约束草案，避免仅依赖前端校验。

### 13.1 服务端权威原则

- 联机模式下，**服务端必须是唯一权威状态源**。
- 客户端提交的所有落子、认输、求和、结算请求，均必须由服务端复算并校验。
- 客户端不得自行决定胜负、计分和合法区域。

### 13.2 服务端至少需要校验的内容

- 当前是否轮到该玩家。
- 请求中的小棋盘编号和格位编号是否合法。
- 目标格位是否为空。
- 当前是否满足投影规则或自由落子规则。
- 当前小棋盘是否已被占领。
- 当前对局是否已结束。
- 主动结算请求是否满足触发条件。
- 求和次数是否超限。

### 13.3 建议的联机事件

| 事件名 | 方向 | 说明 |
|--------|------|------|
| `room:create` | Client → Server | 创建房间 |
| `room:join` | Client → Server | 加入房间 |
| `game:move` | Client → Server | 提交落子 |
| `game:state` | Server → Client | 广播最新对局状态 |
| `game:resign` | Client → Server | 发起认输 |
| `game:draw-offer` | Client → Server | 发起求和 |
| `game:draw-response` | Client → Server | 响应求和 |
| `game:settle` | Client → Server | 请求主动结算 |
| `game:error` | Server → Client | 返回非法操作原因 |

### 13.4 断线重连最低要求

- 房间内任意一方掉线后，服务端应在限定时间内保留房间状态。
- 玩家重连后应恢复：
  - 当前完整棋局状态
  - 当前轮次
  - 合法落子区域
  - 已使用求和次数
  - 待响应求和状态
- 若超出保留时间未重连，可按产品策略判负或判流局。

## 十四、测试与验收补充（新增）

### 14.1 规则测试最低覆盖建议

至少应覆盖以下测试场景：

1. 首手任意落子。
2. 第二手起按投影规则限制落子。
3. 投影目标小棋盘已占领时触发自由落子。
4. 投影目标小棋盘虽未占领但已无可用位置时触发自由落子。
5. 小棋盘三连珠后立即变为占领状态。
6. 已占领小棋盘禁止继续落子。
7. 所有未占领小棋盘下满时自动结算。
8. 所有未占领小棋盘均无法再形成胜方时允许主动结算。
9. 认输后立即结束对局。
10. 单方求和次数超过 3 次后被拒绝。

### 14.2 UI 验收补充

除原有视觉验收外，建议补充以下可操作性验收：

- 玩家能够一眼看出哪里可以下。
- 玩家能够一眼看出上一手下在哪里。
- 玩家能够明确看到当前比分（双方占领小棋盘数）。
- 对非法点击要有明确反馈，而不是静默无响应。
- 移动端下棋盘与按钮区不能互相遮挡。

### 14.3 构建与交付验收补充

- `npm install` 可成功安装依赖。
- `npm run dev` 可成功启动本地开发环境。
- `npm test` 可运行并通过核心规则测试。
- `npm run build` 可成功生成生产构建产物。

## 十五、里程碑重定义（新增）

为避免“目标态规格”和“当前仓库状态”混淆，建议将里程碑拆分为如下阶段：

| 里程碑 | 目标 | 当前状态 |
|--------|------|----------|
| M1 | 本地双人 + 纯规则引擎 + 基础 Web UI | 已完成 |
| M2 | 增强交互（音效、设置、引导完善、更多测试） | 待完成 |
| M3 | 联机房间、服务端权威校验、断线重连 | 待完成 |
| M4 | AI 对战与难度分级 | 待完成 |
| M5 | 3D 渲染升级、材质与动效增强 | 待完成 |
| M6 | 持久化、用户系统、战绩与运维能力 | 待完成 |

### 15.1 M1 完成定义

满足以下条件即可视为 M1 完成：

- 用户可在浏览器中完成一整局本地双人对战。
- 核心规则与规格一致。
- 有基本视觉反馈与操作提示。
- 至少具备自动化规则测试与可执行构建命令。

### 15.2 M2 之后的优先建议

如果后续开发资源有限，建议优先顺序如下：

1. 先补联机服务端权威校验。
2. 再补断线重连与房间生命周期管理。
3. 然后补 AI 与音效等增强体验项。
4. 最后升级完整 3D 表现。
