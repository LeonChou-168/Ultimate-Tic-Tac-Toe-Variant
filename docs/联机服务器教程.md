# 联机服务器教程

本文档说明如何在当前项目基础上增加一个可用的联机对战服务器。目标不是一次性做完完整生产系统，而是先做出一个**规则可信、前后端可联调、后续易扩展**的版本。

---

## 1. 当前项目现状

当前仓库已经具备这些基础：

- React + Vite 前端
- `src/game/engine.ts` 纯 TypeScript 游戏规则引擎
- `src/game/types.ts` 状态类型定义
- `src/game/engine.test.ts` 核心规则测试

这意味着你已经有了一个很好的联机基础：

> **服务端不需要重新实现一套规则。**
> 最好的做法是让服务端直接复用同一套 `engine.ts`，由服务端作为权威状态源。

---

## 2. 推荐架构

建议采用下面这种结构：

```text
Ultimate-Tic-Tac-Toe-Variant/
├── src/                     # 现有前端
├── server/
│   ├── index.ts             # 服务端入口
│   ├── rooms.ts             # 房间管理
│   ├── types.ts             # 联机事件类型
│   └── game-session.ts      # 单局状态封装
├── shared/
│   └── game/                # 可选：后续把前后端共用逻辑迁到这里
└── package.json
```

### 为什么这样设计

- 前端负责展示和交互。
- 服务端负责房间、回合同步、合法性校验、防作弊。
- 游戏规则以服务端计算结果为准。
- 客户端只发送“我想下哪一步”，不能自己决定最终状态。

---

## 3. 推荐技术选型

按照你现在的项目，最省力的联机方案是：

- **Node.js**：与前端同语言，方便复用规则引擎
- **Socket.io**：房间、重连、事件收发更顺手
- **TypeScript**：和现有代码保持一致

先安装依赖：

```bash
npm install socket.io express
npm install -D tsx
```

如果你想把服务端也纳入类型检查，后面可以继续扩展 `tsconfig`。

---

## 4. 第一步：创建最小服务端

新建 `server/index.ts`：

```ts
import express from 'express';
import { createServer } from 'node:http';
import { Server } from 'socket.io';

const app = express();
const httpServer = createServer(app);

const io = new Server(httpServer, {
  cors: {
    origin: 'http://localhost:5173',
    methods: ['GET', 'POST'],
  },
});

app.get('/health', (_req, res) => {
  res.json({ ok: true });
});

io.on('connection', (socket) => {
  console.log('client connected:', socket.id);

  socket.on('disconnect', () => {
    console.log('client disconnected:', socket.id);
  });
});

httpServer.listen(3001, () => {
  console.log('socket server running at http://localhost:3001');
});
```

然后在 `package.json` 里加一个脚本：

```json
{
  "scripts": {
    "server:dev": "tsx server/index.ts"
  }
}
```

启动方式：

```bash
npm run server:dev
```

这一步的目标只有一个：**让联机服务先跑起来。**

## 5. 第二步：设计房间模型

联机井字棋最核心的是“房间”和“权威状态”。建议每个房间维护：

```ts
type RoomPlayer = {
  socketId: string;
  player: 'black' | 'white';
};

type RoomState = {
  roomId: string;
  players: RoomPlayer[];
  gameState: GameState;
};
```

其中 `gameState` 直接使用你现有 `src/game/types.ts` 里的 `GameState`。

建议服务端维护：

```ts
const rooms = new Map<string, RoomState>();
```

### 房间最少需要支持的能力

- 创建房间
- 加入房间
- 广播最新棋局
- 玩家断开处理
- 游戏结束处理

---

## 6. 第三步：定义 Socket 事件

推荐先做这一组最小事件：

### 客户端 -> 服务端

- `room:create`
- `room:join`
- `game:move`
- `game:resign`
- `game:offer-draw`
- `game:respond-draw`
- `game:settle`

### 服务端 -> 客户端

- `room:created`
- `room:joined`
- `room:update`
- `game:update`
- `game:error`
- `game:ended`

---

## 7. 第四步：服务端必须成为规则裁判

这是联机实现里最重要的一条：

> 客户端只能发送意图，服务端负责调用 `placeMove`、`resignGame`、`offerDraw`、`respondToDraw`、`settleGame`。

也就是说，客户端不要自己修改棋盘后再“同步给服务端”，而是：

1. 客户端点击某格
2. 发送 `{ roomId, boardIndex, cellIndex }`
3. 服务端找到对应房间
4. 服务端校验当前 socket 是否轮到该玩家
5. 服务端调用 `placeMove(room.gameState, boardIndex, cellIndex)`
6. 如果成功，更新房间状态并广播
7. 如果失败，返回错误消息

示例：

```ts
socket.on('game:move', ({ roomId, boardIndex, cellIndex }) => {
  const room = rooms.get(roomId);
  if (!room) {
    socket.emit('game:error', { message: '房间不存在' });
    return;
  }

  const current = room.players.find((player) => player.socketId === socket.id);
  if (!current) {
    socket.emit('game:error', { message: '你不在该房间中' });
    return;
  }

  if (current.player !== room.gameState.currentPlayer) {
    socket.emit('game:error', { message: '当前不是你的回合' });
    return;
  }

  const result = placeMove(room.gameState, boardIndex, cellIndex);
  if (!result.ok) {
    socket.emit('game:error', { message: result.message });
    return;
  }

  room.gameState = result.state;
  io.to(roomId).emit('game:update', {
    state: room.gameState,
    message: result.message,
  });
});
```

## 8. 第五步：前端如何接入

前端建议新增一个“模式选择”：

- 本地模式
- 联机模式

本地模式继续使用现在的 `useState(createInitialState())`。

联机模式则改成：

- 页面初始化时连接 Socket.io
- 从服务端接收最新 `GameState`
- 所有点击行为改为发送事件，而不是本地直接调用 `placeMove`

### 推荐做法

把前端状态分成两层：

1. **本地 UI 状态**：按钮 loading、房间号输入框、连接状态
2. **服务端棋局状态**：由 `game:update` 推送下发

这样不会把“联机连接逻辑”和“游戏规则”混在一起。

---

## 9. 第六步：求和、认输、主动结算如何联机化

你现有引擎已经有这些能力，所以联机化主要是把它们包装成服务端事件。

### 认输

客户端发送：

```ts
socket.emit('game:resign', { roomId });
```

服务端执行：

- 校验玩家属于房间
- 调用 `resignGame(room.gameState, player)`
- 广播 `game:update` 或 `game:ended`

### 求和

客户端发送：

```ts
socket.emit('game:offer-draw', { roomId });
```

对方客户端收到后展示“接受 / 拒绝”。

### 主动结算

客户端发送：

```ts
socket.emit('game:settle', { roomId });
```

服务端调用 `settleGame`；如果当前不满足条件，就回 `game:error`。

## 11. Railway / Vercel 部署建议

如果你希望把联机版本放到公网，推荐采用下面这种部署拆分：

- **前端**：Vercel
- **联机后端**：Railway

原因很简单：

- Vercel 很适合部署 Vite/React 前端
- Railway 更适合部署当前这种 `Socket.io + Express` 常驻连接服务

### 推荐环境变量

后端：

```text
CLIENT_ORIGIN=https://your-frontend-domain.vercel.app
ROOM_TTL_MS=1800000
DISCONNECT_GRACE_MS=300000
```

前端：

```text
VITE_SOCKET_SERVER_URL=https://your-railway-domain.up.railway.app
```

### 更详细的公网部署步骤

请继续参考：

- [`Railway 联机部署指南.md`](Railway%20联机部署指南.md)
- [`Vercel 前端部署指南.md`](Vercel%20前端部署指南.md)

- 标签页 A：创建房间
- 标签页 B：加入房间
- A/B 轮流落子，观察服务端广播是否正常

### 重点联调内容

- 非当前回合能否被服务端拒绝
- 非法位置落子能否被服务端拒绝
- 小棋盘占领后是否同步锁定
- 投影目标是否一致
- 主动结算条件是否与单机一致
- 一方断线后房间如何处理

## 11. 第八步：部署建议

开发阶段可以前后端分开：

- 前端：Vercel / Netlify
- 服务端：Railway / Render / Fly.io / 云服务器

### 部署时要注意

1. Socket.io 服务端要开放 WebSocket 支持
2. 前端连接地址不要再写死 `localhost`
3. CORS 白名单要配置为你的前端域名
4. 如果后面要支持重连，房间状态最好放 Redis，而不是只放进程内存

## 12. 第九步：从 demo 到正式服务的升级路线

如果你只是先做一个能玩的联机版本，内存房间就够了。

如果你要往正式产品走，建议逐步补这些能力：

### 第一阶段：可联调 demo

- 内存房间
- 双人加入
- 服务端权威规则
- 落子 / 认输 / 求和 / 主动结算

### 第二阶段：稳定性

- 断线重连
- 房间超时回收
- 更清晰的错误码
- 服务端日志

### 第三阶段：可部署产品

- Redis 房间状态
- 用户系统
- 战绩记录
- 匹配系统
- 观战与回放

---

## 13. 最推荐的下一步

如果你想最快落地联机功能，我建议按这个顺序做：

1. 新建 `server/index.ts`
2. 跑通 Socket.io 服务
3. 做 `room:create` / `room:join`
4. 服务端接管 `game:move`
5. 前端接收 `game:update`
6. 再接认输、求和、主动结算

这个顺序的好处是：

- 每一步都能单独验证
- 出错时容易定位
- 不会一下子把 UI、服务端、规则全搅在一起

---

## 14. 补充建议

为了后面联机更顺，建议你再做两件事：

### 1）把共享逻辑逐步集中

如果服务端也要直接用规则引擎，后面可以把：

- `src/game/engine.ts`
- `src/game/types.ts`

迁移到一个前后端共享目录，例如：

```text
shared/game/
```

这样不会出现一份前端规则、一份后端规则的分叉问题。

### 2）给联机事件补类型

可以单独建一个 `server/types.ts` 或 `shared/socket-types.ts`，把所有事件 payload 写成显式类型，后面维护会轻松很多。
